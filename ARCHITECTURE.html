04 Data Architecture & Scalability

Normalization by Reference (Dual-Copy Pattern)

Parvah Connect uses a denormalized reference strategy — every hierarchy exists in two locations: a per-user copy (tracks personal status like draft/pending/active) and a global copy (source of truth for membership, admin lists, and aliases). This is intentional:

Pattern How It Works Why It's Optimal
Per-User Copy
/users/{email}/hierarchies/{orgId}		 Stores the user's personal view: status (draft → pending → active), join flags, timestamps O(1) reads for "my orgs" — no need to scan global data. Each user's dashboard loads only their own sub-collection
Global Source of Truth
/hierarchies/{orgId}		 Stores canonical membership arrays, admin emails, name aliases, and the full sub-node tree Single document read for membership checks. Firestore security rules reference this directly for RBAC enforcement
Sync on Mutation Every create/edit/delete writes to both copies atomically Ensures consistency without expensive cross-collection joins. Firestore transactions guarantee atomicity

Sub-Collection Nesting

Hierarchies use Firestore's native sub-collection model: /hierarchies/{orgId}/sub-nodes/{nodeId}/sub-nodes/{childId}/.... This isn't just organizational — it has critical scalability implications:

O(1)
Document Lookups
Auto
Shard Distribution
1 MB
Max Doc Size (enforced)
∞
Sub-Collection Depth

Why This Scales

Design Choice Scalability Benefit
Sub-collections auto-shard Firestore automatically distributes sub-collections across storage nodes. Adding 10,000 orgs doesn't slow any single org's reads
Status-based collection splitting /problems, /solvedProblems, /archivedProblems are separate collections — distributes hot read/write paths and prevents index bloat
Array membership checks members[] arrays are bounded by org size (typically <1000), not platform size. Security rule checks are O(k) where k = org members
Document-level atomicity Firestore transactions lock at document granularity, not collection. Concurrent operations on different orgs never contend
Denormalized reads Dashboard loads require 1-3 document reads, not complex joins across collections. Read latency stays constant regardless of total data volume
⚠️ Key insight: The dual-copy pattern trades slightly more writes (2x per mutation) for dramatically fewer reads (O(1) per user dashboard load). In Firestore's pricing model, reads outnumber writes ~100:1 in typical usage — making this pattern both faster and cheaper at scale.

14 Security Architecture

Access control is enforced at the Firestore level via security rules:

Collection Read Write Delete
/users/{email} Owner only Owner only Owner only
/hierarchies/{orgId} Any signed-in Members / Admins Org Admin / Super
/pendingApprovals/{id} Super / Assigned / Requester Signed-in (create) Super / Assigned
/problems/{id} Signed-in Owner / Admin / Super Owner / Super
/superAdmin/** Super Admin only Super Admin only Super Admin only

Function Check
